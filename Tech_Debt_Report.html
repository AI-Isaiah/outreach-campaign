Prioritized Tech Debt

Scored as Priority = (Impact + Risk) × (6 − Effort). Higher = fix sooner.

1. cli.py God-File (41 KB, ~1,200+ lines)
CodeCRITICAL · 50
Impact: 5Risk: 5Effort: 4
src/cli.py is by far the largest file in the project at 41 KB — roughly 3× bigger than the next largest service. It almost certainly mixes CLI parsing, business logic, and orchestration into one monolith. This makes it fragile to change, hard to test in isolation, and a constant merge-conflict magnet.
Business case: Every new feature or bug fix risks breaking unrelated CLI workflows. Splitting into focused command modules will accelerate development and reduce regressions.
2. Zero Frontend Tests
TestCRITICAL · 50
Impact: 5Risk: 5Effort: 4
The React frontend has 31 source files totaling ~183 KB including pages like Pipeline (20 KB), ContactDetail (17 KB), and complex components like QueueLinkedInCard (13 KB) — yet there are zero test files. Any refactor or feature change in the frontend is completely unguarded.
Business case: The frontend is the user-facing layer. Without tests, every deployment is a roll of the dice. Adding Vitest + React Testing Library for critical paths (Pipeline, Queue, ContactDetail) provides an immediate safety net.
3. SQLite Database Still in Repo (1.4 MB)
ArchitectureInfraHIGH · 40
Impact: 4Risk: 4Effort: 2
The project has migrated to PostgreSQL (7 pg migrations, Supabase config, migrate script), yet outreach.db (1.4 MB), outreach.db-shm, and outreach.db-wal are still sitting in the root directory. The .gitignore blocks *.db from commits, but the files are already checked in and still on disk. There are also duplicate SQLite-era migrations (migrations/001_initial_schema.sql and 002_web_app.sql) alongside the proper migrations/pg/ directory.
Business case: Stale SQLite artifacts confuse new developers ("which database am I using?"), risk accidental reads from the wrong store, and bloat the repo. Quick cleanup.
4. No Authentication / Authorization
ArchitectureHIGH · 36
Impact: 4Risk: 5Effort: 3.5
The FastAPI web app and 16 route files expose a full CRM (contacts, deals, campaigns, emails, WhatsApp data) without any visible auth middleware. There's a specs/02_SECURITY_AUTH.md spec (11 KB) that appears to plan for this — but it hasn't been implemented. A .env with real credentials exists on disk.
Business case: This is a compliance and data protection risk. PII (names, emails, phone numbers, AUM data) is accessible to anyone who can reach the API. Must be addressed before any multi-user or public deployment.
5. Five Backend Services Have No Tests
TestHIGH · 32
Impact: 4Risk: 4Effort: 3
Missing test coverage for: ab_testing.py, compliance.py, gmail_drafter.py, linkedin_actions.py, template_engine.py. The compliance module in particular handles GDPR/unsubscribe logic — the kind of code that absolutely must be regression-tested. Web API routes also have only a single test file (test_web_api.py) covering 16 route modules.
Business case: Untested compliance logic is a legal risk. Untested email/LinkedIn integration code can send incorrect messages to real contacts.
6. Scattered Documentation / Spec Debt
DocumentationHIGH · 30
Impact: 3Risk: 3Effort: 2
The root directory contains 10+ large markdown files (FINAL_HANDOFF 55 KB, Implementation_Plan 52 KB, CODE_REVIEW 24 KB, ENGINEERING_REVIEW 12 KB, etc.) alongside 6 spec files in specs/ and more in docs/. 7. Oversized Frontend Components
CodeMEDIUM · 24
Impact: 3Risk: 3Effort: 3
Several React components are very large: Pipeline.tsx (20 KB), ContactDetail.tsx (17 KB), QueueLinkedInCard.tsx (13 KB), QueueEmailCard.tsx (12 KB). These likely mix data fetching, business logic, and UI rendering in single files — a maintenance burden and a barrier to reuse.
Business case: Large components are harder to modify, slower to review, and prone to state bugs. Breaking them into container/presenter pairs with custom hooks improves velocity.
8. No API Client Type Safety
CodeArchitectureMEDIUM · 24
Impact: 3Risk: 3Effort: 3
The frontend has a single api/client.ts (11 KB) and a types/index.ts (7 KB) but the backend is plain Python with no OpenAPI schema generation. The frontend types are likely manually maintained — any backend schema change can silently break the frontend.
Business case: Backend-frontend contract drift causes runtime errors. Adding FastAPI response models + auto-generated TypeScript types eliminates an entire class of bugs.
9. Migration Management — No Runner, Dual Formats
InfraMEDIUM · 20
Impact: 2Risk: 4Effort: 3
Migrations live in two directories (migrations/ for old SQLite, migrations/pg/ for Postgres) with no migration runner or version tracking in code. The database.py module is only 2 KB, suggesting migrations might be run manually. No rollback support visible.
Business case: Without versioned, automated migrations, schema changes in production are risky. Adopt Alembic or a simple version-tracked runner.
