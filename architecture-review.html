2. What's Working Well

Clean dependency flow

Routes → Services → Models → DB. Services never import from web layer. No circular dependencies (one deferred import in linkedin_actions).

Services are pure & testable

All services accept conn as first param. No global state, no side effects beyond DB writes. Can be unit-tested with a test database.

Lightweight CRUD layer

Raw psycopg2 with RealDictCursor — no ORM overhead. SQL is explicit and optimizable. Window functions in priority_queue work cleanly.

Frontend: React Query as data layer

Server state cleanly separated from UI state. No Redux needed. Stale-while-revalidate, cache invalidation on mutations, lazy page loading all in place.

Type safety throughout

Frontend: TypeScript strict mode, all API responses typed (635 LOC of types). Backend: Pydantic request models in FastAPI routes.

Defensive migration pattern

Migrations run on every command/startup. CREATE TABLE IF NOT EXISTS makes them idempotent. No external migration tool needed.

3. Structural Issues

3.1 Routes contain business logic

Route handlers in queue.py (140+ lines) do batch fetching, template rendering, AUM filtering, and firm-type grouping. This logic should live in an application service layer, not the HTTP handler. It can't be reused from CLI commands.

Impact: Code duplication between CLI and web, routes are hard to test, business logic tied to HTTP context.

3.2 Monolithic CRUD module

models/campaigns.py is 1,000+ lines handling campaigns, templates, contacts, events, enrollments, and sequence steps. Finding a function requires scrolling through unrelated CRUD.

Impact: Navigation friction, merge conflicts, cognitive load.

3.3 No application service / orchestration layer

Routes call services directly, sometimes composing 3-4 service calls with their own transaction logic. There's no intermediate layer to orchestrate multi-service operations.

Impact: Routes become fat, transaction boundaries are ad-hoc, CLI and web can't share orchestration logic.

3.4 No background job system

All operations are request-scoped. Gmail scanning, bulk email, AI analysis, and metric computation must be triggered by a user action or external cron. Long-running tasks block the web server.

Impact: Limits scalability, no scheduled automation, poor UX for slow operations.

3.5 Frontend: flat component directory at scale

26 components in a flat /components/ folder with no feature-based grouping. API client is 572 lines with 100+ methods in a single flat object. No query key factory, no constants file.

Impact: Hard to navigate, duplicate constants in 4+ pages, magic strings for query keys.

4. Architecture Decision Records

ADR-001: Introduce Application Service Layer
Proposed

Date: 16 Mar 2026  |  Deciders: Backend lead

Context

Route handlers in queue.py, contacts.py, and deals.py perform multi-service orchestration (batch fetching, template rendering, compliance checks, state transitions). This logic can't be reused from CLI commands. CLI commands duplicate some of this orchestration in src/commands/.

Decision

Create src/application/ with orchestration services that compose domain services. Routes become thin HTTP adapters; CLI commands call the same application services.

PROPOSED STRUCTURE src/application/ ├── queue_app.py # get_daily_queue() — composes priority_queue + template_engine + compliance ├── contact_app.py # create_contact() — composes normalization + dedup + enrollment ├── import_app.py # import_csv() — composes import + dedup + verification └── campaign_app.py # send_batch() — composes email_sender + state_machine + metrics Routes: route → app_service → [service₁, service₂, ...] → models → DB CLI: command → app_service → [service₁, service₂, ...] → models → DB

Options Considered

DimensionA: App Service LayerB: Keep Routes FatC: Move Logic to Services
ComplexityMedium (new layer)Low (status quo)Medium (services grow)
Code reuse CLI↔WebFullNonePartial
TestabilityHigh — test app servicesLow — must mock HTTPMedium
Transaction clarityExplicit in app serviceAd-hoc in routesImplicit

Trade-off Analysis

Option A adds a thin layer (~500 lines) but eliminates ~300 lines of duplicated orchestration between CLI and web. Transaction boundaries become explicit. The downside is indirection — developers must look in application/ for business logic instead of routes.

Consequences

Easier: Testing orchestration logic, reusing logic from CLI, understanding transaction boundaries.
Harder: Tracing a request through one more layer, deciding what's "orchestration" vs "service logic".
Revisit: If the app service layer grows beyond 6-8 files, consider whether some should merge back into domain services.

Action Items

1. Create src/application/queue_app.py by extracting from routes/queue.py (highest LOC).
2. Move batch-fetch + render logic from queue route into app service.
3. Update CLI queue command to call app service.
4. Repeat for contacts, campaigns. Estimated effort: 6-8 hours.

ADR-002: Split CRUD Layer into Domain Repositories
Proposed

Date: 16 Mar 2026  |  Deciders: Backend lead

Context

models/campaigns.py is 1,000+ lines containing CRUD for campaigns, templates, contacts, events, enrollments, and sequence steps. Finding a function requires scrolling through unrelated operations. The file is a frequent source of merge conflicts.

Decision

Split into focused repository modules with a shared base pattern for cursor management.

PROPOSED STRUCTURE src/repositories/ ├── base.py # execute_query(), fetch_one(), fetch_all() helpers ├── campaign_repo.py # campaigns, sequence_steps (~200 LOC) ├── template_repo.py # templates, variant management (~150 LOC) ├── contact_repo.py # contacts, enrollment, status (~250 LOC) ├── event_repo.py # events, dedup_log (~150 LOC) └── company_repo.py # companies (~100 LOC)

Options Considered

DimensionA: Split RepositoriesB: Add ORM (SQLAlchemy)C: Keep Monolith
Migration effort4 hrs (move + re-import)20+ hrs (rewrite all SQL)0 hrs
SQL flexibilityFull — same raw SQLReduced — ORM abstractsFull
DiscoverabilityHigh — domain-scoped filesHigh — model classesLow — 1000 lines
Cursor boilerplateEliminated via base.pyEliminated via sessionRepeated 40+ times

Trade-off Analysis

Option A is the lowest-risk change — it's a pure refactor (move functions to new files, update imports). Option B (ORM) would be cleaner long-term but is a massive rewrite with risk of breaking complex SQL (window functions, CTEs). Option C is free but the file will only grow.

Consequences

Easier: Finding CRUD functions, reducing merge conflicts, writing focused tests.
Harder: Import paths change (one-time cost to update all callers).
Revisit: If repository count exceeds 8, consider consolidation.

Action Items

1. Create src/repositories/base.py with fetch_one(), fetch_all(), execute() helpers.
2. Split campaigns.py into 5 repository files, keeping existing function signatures.
3. Update all imports across services and routes (use IDE refactor).
4. Re-export from models/campaigns.py for backwards compatibility during transition. Estimated effort: 4 hours.

ADR-003: Add Background Job System
Proposed

Date: 16 Mar 2026  |  Deciders: Backend lead, DevOps

Context

All operations are request-scoped. Gmail reply scanning (10-30s), bulk email sends (variable), AI analysis (60s+), and metric computation block the web server. Scheduled tasks require external cron. The research job system already needs polling for status updates.

Decision

Add a lightweight task queue. Start with APScheduler + in-process workers for simplicity. Migrate to Celery + Redis only if task volume exceeds ~100/day.

Options Considered

DimensionA: APScheduler (in-process)B: Celery + RedisC: External Cron
Setup complexityLow — pip install, configMedium — Redis infra neededLow — crontab
Scalability~100 tasks/dayThousands/dayManual only
Job visibilityLogs onlyFlower dashboardNone
Retry/dead-letterBasicFullNone
Deployment changeNone (same process)+1 worker, +RedisNone

Trade-off Analysis

APScheduler runs inside the FastAPI process, avoiding infrastructure changes. It handles the 4-5 scheduled tasks (reply scan, metric refresh, daily queue prep, email batch). If task volume grows or we need distributed workers, migrate to Celery — the task functions stay the same, only the dispatch changes.

ADR-004: Reorganize Frontend into Feature Modules
Proposed

Date: 16 Mar 2026  |  Deciders: Frontend lead

Context

The frontend has 18 pages, 26 components, a 572-line API client, and a 635-line types file — all in flat directories. Constants (lifecycle stages, status colors) are duplicated in 4+ pages. Query keys are magic strings. As the app grows, this structure becomes harder to navigate.

Decision

Reorganize into feature modules with shared infrastructure. Don't move to a monorepo or framework — just restructure within the existing Vite + React setup.

PROPOSED STRUCTURE frontend/src/ ├── api/ │ ├── client.ts # Base request() function only (~50 LOC) │ ├── queue.ts # Queue-specific endpoints │ ├── contacts.ts # Contact endpoints │ ├── campaigns.ts # Campaign endpoints │ └── index.ts # Re-exports all APIs ├── constants/ │ ├── statuses.ts # ContactStatus, LifecycleStage unions │ ├── queryKeys.ts # Typed query key factory │ └── colors.ts # Status→color mappings ├── hooks/ │ ├── useForm.ts # Generic form hook │ ├── useModal.ts # Modal open/close state │ └── usePagination.ts # Shared pagination logic ├── components/ │ ├── shared/ # Badges, buttons, DataTable, skeleton │ ├── queue/ # QueueEmailCard, QueueLinkedInCard │ ├── contacts/ # ContactForm, ContactTimeline │ └── pipeline/ # StageColumn, DealCard, PipelineFilters ├── pages/ # Thin page shells (import from features) └── types/ ├── contacts.ts # Contact, Company types ├── queue.ts # QueueItem, QueueResponse └── index.ts # Re-exports

Options Considered

DimensionA: Feature ModulesB: Full Feature SlicesC: Keep Flat
Migration effort6 hrs (move + update imports)16 hrs (restructure everything)0 hrs
DiscoverabilityGood — domain foldersBest — self-containedDeclining
Refactoring riskLow — moves onlyMedium — new patternsNone
Scales to 50+ componentsYesYesNo

Trade-off Analysis

Option A is a series of safe file moves. It doesn't introduce new patterns (no barrel imports required, no circular dependency risk). Option B (full feature slices where each feature owns its own types, hooks, API) is cleaner but requires more upfront work and discipline. Option C is free but the flat structure degrades as components grow past 30-40.

Action Items

1. Create constants/queryKeys.ts + constants/statuses.ts — immediate dedup win.
2. Split api/client.ts into domain files (queue, contacts, campaigns, etc.).
3. Move feature-specific components into subdirectories.
4. Extract useForm hook from modal patterns. Estimated effort: 6 hours.

7. Architecture Principles Going Forward

Dependency Rule

Dependencies always point inward: Routes → App Services → Domain Services → Repositories → DB. Never import outward.

Transaction Ownership

Application services own transaction boundaries. Domain services never call conn.commit(). Routes never call services directly for multi-step operations.

Single Responsibility per File

No file exceeds 400 lines. When it does, split by domain or responsibility. God modules are the #1 maintainability killer.

Constants Over Strings

All status values, channel types, and lifecycle stages must come from a shared constants module. No magic strings in business logic.
